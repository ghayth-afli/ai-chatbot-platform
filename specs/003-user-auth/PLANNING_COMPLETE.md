## Design Decisions Summary

### 1. Authentication Architecture

- **Method**: Email/Password + Google OAuth
- **Token Strategy**: JWT with HTTP-only cookies
- **Access Token TTL**: 15 minutes
- **Refresh Token TTL**: 7 days
- **Cookie Flags**: HttpOnly, Secure, SameSite=Lax

### 2. Database Models

**New Models**:

- `VerificationCode` — Stores email verification/reset codes (6-digit, 10-min TTL)

**Extended Django User Model**:

- `auth_provider` — Track authentication method (email/google)
- `profile_picture_url` — Store Google profile picture URL
- `is_verified` — Flag for email verification status

### 3. API Design

**9 REST Endpoints**:

1. `POST /api/auth/signup` — Register new user
2. `POST /api/auth/verify-email` — Verify email with code
3. `POST /api/auth/login` — Email/password authentication
4. `POST /api/auth/logout` — Clear authentication
5. `POST /api/auth/forgot-password` — Send password reset code
6. `POST /api/auth/reset-password` — Reset password with code
7. `POST /api/auth/google` — Google OAuth token exchange
8. `GET /api/auth/me` — Get current user info
9. `POST /api/auth/refresh` — Refresh access token

**Error Handling**:

- Standardized response format with code + details
- 15 error types mapped to user-friendly messages
- i18n support for EN/AR languages

### 4. Frontend Architecture

**Protected Routes**: AuthGuard component wraps authenticated pages  
**Auth State**: `useAuthStatus` hook syncs with backend  
**axios Interceptors**: Auto-attach token, auto-refresh on 401  
**Form Validation**: Per-field errors with i18n translations

### 5. Security Implementation

- Password hashing: bcrypt (PBKDF2 fallback)
- Rate limiting: DRF throttling classes (built-in)
  - Login: 5 attempts per 15 minutes
  - Code requests: 3 per 60 minutes
- CSRF protection: Automatic with cookies + SameSite
- Email verification: 6-digit code (10-min expiration)
- OAuth: Backend token exchange (secure)

### 6. Internationalization (EN/AR)

- All UI pages support EN/Arabic with RTL layout
- Email templates bilingual (verify_email.html + verify_email.ar.html)
- Error messages translated via i18next
- Language preference stored in user profile

## Key Technical Insights

### 1. JWT Tokens Configuration

- SIMPLE_JWT library already installed but not configured
- Settings need explicit JWT configuration (added to plan.md)
- Both access and refresh tokens use secure HTTP-only cookies

### 2. User Model Extension

- Django's native User model used (username = email for simplicity)
- Custom Profile model (1:1) stores language preference
- New auth_provider and is_verified fields added via migration

### 3. Email Verification Strategy

- 6-digit random codes are standard (1M combinations, SMS-friendly)
- 10-minute TTL prevents code reuse from interception
- Previous codes auto-invalidated when new code requested (rate limit enforcement)

### 4. Google OAuth Implementation

- Backend token exchange (secure, no frontend token handling)
- User auto-created on first login with Google data (email, name, picture)
- OAuth tokens discarded after exchange (security best practice)

### 5. Frontend Authentication Flow

- axios interceptor auto-refreshes access token on 401
- useAuthStatus hook checks auth on app init (GET /api/auth/me)
- Protected routes wrapped with AuthGuard component
- No localStorage tokens (HTTP-only cookies only)

## Known Constraints & Future Enhancements

### Phase 3 Scope (Current)

- ✅ Email/password authentication
- ✅ Email verification
- ✅ Password reset
- ✅ Google OAuth
- ✅ Cookie-based sessions

### Out of Scope (Phase 4+)

- ❌ 2FA / MFA authentication
- ❌ Account linking (one email → multiple auth providers)
- ❌ Social providers beyond Google (Facebook, GitHub, etc.)
- ❌ OAuth to the nexus platform (API for third-party apps)

- ❌ Native mobile app support
